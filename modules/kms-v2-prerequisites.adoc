= Prerequisites for KMS encryption

[role="_abstract"]
Before you configure KMS encryption, you must prepare your cluster, deploy an external KMS provider, and ensure network connectivity between control plane nodes and the KMS provider.

== Cluster requirements

* You have access to the cluster as a user with the `cluster-admin` role.
* The `TechPreviewNoUpgrade` feature gate is acceptable for your environment. This feature gate blocks cluster upgrades while KMS encryption is enabled.
* You have created a recent backup of the etcd database.
* You have access to all control plane nodes to deploy the KMS plugin.

== External KMS provider requirements

* A KMS v2-compatible provider is deployed and accessible from control plane nodes.
* The KMS provider supports the Kubernetes KMS v2 protocol specification.
* The KMS provider has sufficient capacity to handle encryption and decryption requests from all API servers.
* The KMS provider is configured with high availability to prevent cluster failures if the provider becomes unavailable.

== Network requirements

* Control plane nodes can establish network connections to the KMS provider endpoint.
* Firewall rules permit outbound connections from control plane nodes to the KMS provider on the required ports.
* DNS resolution is configured for the KMS provider hostname if using a DNS-based endpoint.
* Network latency between control plane nodes and the KMS provider is less than 100ms to avoid API server timeout issues.

== Authentication requirements

Depending on your chosen KMS provider, you must configure one of the following authentication methods:

AWS KMS:: IAM role attached to control plane instances, or AWS access keys with KMS permissions
Azure Key Vault:: Managed identity assigned to control plane VMs, or service principal with Key Vault access
HashiCorp Vault:: AppRole credentials, token, or Kubernetes authentication mount
Custom provider:: Provider-specific authentication as required by the KMS v2 plugin

== Disconnected environment considerations

If you are deploying {product-title} in a disconnected or restricted network environment:

* Cloud-based KMS providers like AWS KMS and Azure Key Vault require internet connectivity or VPC/private endpoints.
* On-premise KMS solutions like self-hosted HashiCorp Vault are better suited for fully disconnected environments.
* KMS plugin container images must be mirrored to your disconnected registry.
* Self-signed certificates for on-premise KMS providers require additional trust configuration.

== Version compatibility

* {product-title} {ocp-nplus1} or later
* KMS provider must support KMS v2 protocol (Kubernetes KMS API v2)
